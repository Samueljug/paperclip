# Plan: Technical Design Before Implementation

## Purpose

The plan sits between understanding a ticket (discover) and writing code. Without it, developers figure things out as they go — which leads to rework, architectural drift, and code that doesn't fit existing patterns. A 15-minute planning step prevents days of rework.

This skill produces a concrete, actionable technical plan: which files to create or modify (in order), what the data model looks like, what the API contracts are, and what tests need to be written. The output is designed to be shared with a senior developer for a 5-minute sanity check before implementation begins.

## When to Use This Skill

- Immediately after the discover skill completes and the developer confirms the analysis (auto-chain)
- When a developer asks how to build or architect something
- When a developer asks what files they need to change
- When a developer wants to break down a task before starting
- Before any implementation work on medium or large tickets

## Relationship to Other Skills and Docs

This skill works best when the project has the modular CLAUDE.md system set up (generated by the `generate-claude-md` skill). If `docs/claude/` reference docs exist, read the relevant ones before planning:

- **Always read** `docs/claude/architecture.md` — to understand the project structure and propose changes that fit
- **Always read** `docs/claude/patterns.md` — to ensure the plan follows existing code patterns
- **Read if API work is involved:** `docs/claude/api-conventions.md` — to design endpoints that match existing conventions
- **Read if database work is involved:** `docs/claude/database.md` — to design schemas and queries that follow existing patterns
- **Read if security-sensitive:** `docs/claude/security.md` — to ensure auth, tenant isolation, and permissions follow existing patterns
- **Read if test planning:** `docs/claude/testing.md` — to plan tests that follow existing test conventions

**Always check for existing Architecture Decision Records:** Use Glob to check if `docs/decisions/INDEX.md` exists. If it does:

1. Read `INDEX.md` to get the summary of all decisions with their area tags
2. Identify which ADRs have area tags relevant to this plan (e.g., if planning an API feature, look for ADRs tagged "API", "Auth", "Data Model")
3. Read ONLY the relevant ADR files — don't load superseded or deprecated ones unless investigating why a decision changed
4. Respect "accepted" ADRs: do not re-evaluate decisions that are already settled
5. If new constraints mean an existing ADR no longer applies, flag this explicitly and propose superseding it (the developer must approve)
   Read `references/adr-guide.md` for the full ADR format, lifecycle, and rules.

If these docs don't exist, extract what you can from the codebase directly (read representative files to understand patterns before planning).

If the discover skill was run earlier in this session, use its output as input — requirements, codebase impact, risk assessment, existing ADRs surfaced, and suggested implementation order should all feed into the plan.

## Inputs

The developer provides one of:

- Confirmation after a discover analysis ("looks good, let's plan" / "ready to proceed")
- A description of what needs to be built
- A ticket reference with enough context to plan against

If the input is too vague to plan (e.g., "plan the thing"), ask the developer for more detail. You need at least: what is being built and what the expected outcome is.

## Planning Process

Work through each section in order. Use Grep, Glob, Read, and Task to explore the codebase thoroughly. Every file path, function name, and pattern reference must be verified against the actual codebase — do not guess.

### Section 1: Approach Summary

Write 3-5 sentences describing the high-level technical approach. This is the elevator pitch — if a senior reads nothing else, they should understand the strategy from this section.

Include:

- What architectural pattern this follows (and reference the existing pattern in the codebase)
- Which layers of the stack are affected (frontend, API, service, database)
- Any key technical decisions (and why)

If there are multiple valid approaches, briefly describe each (2-3 sentences per approach) with tradeoffs, and recommend one. Explain why.

**ADR check before evaluating approaches:**

Before proposing approaches, check if any existing ADR in `docs/decisions/` already covers this decision area. If an ADR exists and still applies:

- State: "This follows the decision in ADR-NNN: [title]. [Brief summary of the decision.]"
- Do NOT re-evaluate alternatives that were already rejected.
- Build your approach on top of the existing decision.

If an existing ADR exists but new constraints make it no longer applicable:

- Flag: "ADR-NNN decided [X], but this ticket introduces new constraints: [what changed]. Recommending this ADR be superseded."
- Evaluate approaches including the new constraints.
- The developer must approve superseding the ADR.

**Auto-generate ADR when a significant decision is made:**

If this section evaluates 2+ genuine approaches with meaningful tradeoffs (not trivial choices), automatically create a new ADR after presenting the recommendation. This captures the decision for future reference.

Significance threshold — create an ADR if:

- Two or more architecturally different approaches were genuinely considered
- The rejected alternatives had real merit (not obviously wrong)
- Future developers working in this area would plausibly face the same decision

Do NOT create an ADR for:

- Choices with one obvious correct answer
- Following an existing pattern (no decision to record)
- Implementation details that don't affect architecture

When creating an ADR:

1. Determine the next ADR number by reading `docs/decisions/INDEX.md` (or checking existing files with Glob if INDEX.md doesn't exist yet)
2. Create `docs/decisions/NNN-[descriptive-slug].md` following the template in `references/adr-guide.md`
3. Set status as **proposed** — NOT accepted. The ADR is part of a plan that hasn't been reviewed yet. Status updates to "accepted" when the implementing PR is merged.
4. Add the entry to `docs/decisions/INDEX.md` (create INDEX.md if it doesn't exist)
5. In the file-by-file plan (Section 7), note which files should include an ADR code comment (e.g., `// Decision: ADR-NNN — see docs/decisions/NNN-slug.md`) at the relevant function or class
6. Keep the ADR under 60 lines
7. Include in the plan output: "📝 **ADR proposed:** `docs/decisions/NNN-[slug].md` — [one-line summary]. Review this with the plan. Status will update to 'accepted' when the PR merges."

If the ADR supersedes an existing one:

1. Set the new ADR's `Supersedes:` field to the old ADR number
2. Update the old ADR's status to `superseded by NNN`
3. Update both entries in INDEX.md
4. Flag in the plan output — superseding a past decision requires explicit developer approval

### Section 2: Data Model

**Skip this section if no database changes are needed.**

For every new or modified schema/model:

1. **Search the codebase** for existing models using Grep and Glob. Read 2-3 existing model definitions to understand the project's schema conventions (field naming, type definitions, index patterns, validation patterns, timestamps).

2. **Define the schema** following the existing conventions exactly. Include:
   - Collection/table name
   - Every field: name, type, required/optional, default value, validation rules
   - Indexes (including compound indexes for common query patterns)
   - Relationships/references to other models
   - Tenant scoping field (if multi-tenant)
   - Timestamps (createdAt, updatedAt, deletedAt if soft-delete)

3. **For modifications to existing models**, show the before and after. Specifically:
   - What fields are being added (with defaults for existing records)
   - What fields are being modified (and whether this is backward-compatible)
   - What indexes are being added or removed
   - Whether a migration script is needed and what it does

4. **Flag any risks:**
   - New required fields without defaults (will break existing records)
   - Index changes on large collections (may lock in production)
   - Field type changes (may require data migration)

### Section 3: API Contracts

**Skip this section if no API changes are needed.**

For every new or modified endpoint:

1. **Search the codebase** for existing route definitions. Read 2-3 existing route handlers to understand: the middleware chain, request validation pattern, response shape, error format, and how auth is applied.

2. **Define each endpoint** following the existing conventions:

```
[METHOD] /api/v1/[resource]

Auth: [auth middleware name] + [permission check if applicable]
Middleware: [list all middleware in order]

Request:
  Params: [URL parameters with types]
  Query: [query parameters with types, defaults, validation]
  Body: {
    field: type (required/optional) — description
  }

Response (200/201):
  {
    field: type — description
  }

Error Responses:
  400: { error: "validation_error", message: "..." } — [when this occurs]
  401: { error: "unauthorized", message: "..." } — [when this occurs]
  403: { error: "forbidden", message: "..." } — [when this occurs]
  404: { error: "not_found", message: "..." } — [when this occurs]
  422: { error: "unprocessable", message: "..." } — [when this occurs]
```

3. **For modifications to existing endpoints**, explicitly flag:
   - Any changed response fields (breaking change for consumers)
   - Any changed request parameters (breaking change for callers)
   - Any changed status codes
   - Whether this needs API versioning

### Section 4: User Experience Definition

**Skip this section if there are no user-facing changes.**

This is where most "it works but it's wrong" bugs come from. The code can be technically perfect and still be wrong if no one specified what the user should actually see and experience. Define the user experience BEFORE the file-by-file plan so the frontend implementation has a clear target.

**4.1 — User flow:**

Walk through the feature step by step from the user's perspective. For each step:

```
Step 1: User [action — what they click, type, or navigate to]
  → They see: [what appears on screen]
  → They can: [what actions are available]

Step 2: User [next action]
  → They see: [result]
```

Keep it concrete. Not "user selects a template" but "user clicks the 'New Document' button in the top-right of the dashboard → a modal opens showing a grid of template cards filtered to their jurisdiction, with a search bar at the top."

**4.2 — UI states:**

For every screen or component involved, define what the user sees in each state:

```
Loading: [what the user sees while data is being fetched — skeleton, spinner, etc.]
Empty: [what the user sees when there's no data — empty state message, CTA]
Populated: [the normal state with data]
Error: [what the user sees when something goes wrong — error message, retry option]
Partial: [what happens if some data loads but other data fails]
```

**4.3 — Validation and feedback:**

For every user input (form fields, selections, uploads):

```
Field: [name]
  Valid: [what constitutes valid input]
  Invalid: [what's rejected and the exact error message shown]
  When validated: [on blur? on submit? real-time?]
```

For every user action (save, delete, submit):

```
Action: [name]
  Success feedback: [what the user sees — toast, redirect, inline confirmation]
  Failure feedback: [what the user sees — error message, which fields are highlighted]
  Loading feedback: [button disabled? spinner? progress bar?]
```

**4.4 — Permissions in the UI:**

What does the user see if they DON'T have permission?

- Is the feature/button hidden entirely? (they don't know it exists)
- Is it visible but disabled? (they know it exists but can't use it)
- Is it accessible but returns an error? (bad UX — avoid this)
- Does the UI adapt based on their subscription tier?

### Section 5: Business Rules Specification

**Skip this section if the feature has no business logic (pure UI or infrastructure changes).**

This section defines the actual business rules BEFORE writing code or tests. You can't test business logic correctly if you haven't specified what "correct" means. Define every business rule as a concrete input → expected output mapping.

For every function that implements a business rule (calculations, conditional logic, pricing, permissions, status transitions, eligibility checks):

```
Rule: [name of the business rule in plain English]
Function: [planned function name and file path]

| Scenario | Input | Expected Output | Notes |
|---|---|---|---|
| Standard case | { seats: 50, plan: 'pro', region: 'AU' } | { subtotal: 2500, gst: 250, total: 2750 } | |
| Boundary - at tier limit | { seats: 30, plan: 'starter' } | { subtotal: 900, gst: 90, total: 990 } | 30 is max for starter |
| Boundary - over tier limit | { seats: 31, plan: 'starter' } | Error: "Starter plan supports up to 30 seats" | Must upgrade to pro |
| Zero/empty | { seats: 0 } | Error: "At least 1 seat required" | |
| Missing required field | { plan: 'pro' } | Error: "Seats is required" | |
| Invalid type | { seats: "abc" } | Error: "Seats must be a number" | |
```

**Why this matters:** These tables become the direct source of truth for the business logic tests in Section 8. Every row in this table becomes a test case. If the developer implements the function and it doesn't produce these exact outputs for these exact inputs, the test fails. No ambiguity.

**State transitions** (if the feature involves status changes, workflows, or lifecycle events):

```
Rule: [what triggers the transition]

| Current State | Event | New State | Side Effects |
|---|---|---|---|
| draft | user clicks "Submit" | pending_review | Email sent to reviewer |
| pending_review | reviewer approves | active | Notification sent to author |
| pending_review | reviewer rejects | draft | Rejection reason saved, author notified |
| active | user clicks "Archive" | archived | Removed from active listings |
| archived | user clicks "Restore" | draft | Returns to draft, not active |
```

**Permission rules** (if the feature has role-based access):

```
| Action | Admin | Firm Owner | Solicitor | Paralegal | Notes |
|---|---|---|---|---|---|
| Create template | ✓ | ✓ | ✗ | ✗ | |
| Edit own template | ✓ | ✓ | ✓ | ✗ | Solicitors can only edit their own |
| Edit any template | ✓ | ✓ | ✗ | ✗ | |
| Delete template | ✓ | ✓ | ✗ | ✗ | Soft delete only |
| View templates | ✓ | ✓ | ✓ | ✓ | Filtered by jurisdiction and tier |
```

### Section 6: Security Design

Every planned change must be evaluated for security at design time, not after the code is written. This section defines the security requirements upfront so they're built in, not bolted on.

**6.1 — Authentication and authorisation:**

For every new or modified endpoint, specify:

```
Endpoint: [METHOD /path]
  Auth required: [yes/no — almost always yes]
  Auth middleware: [exact middleware name from the codebase]
  Permission level: [which roles can access this]
  Ownership check: [does the user need to own the resource, or just be in the same tenant?]
```

Search the codebase (Grep for auth middleware, permission decorators, or guard patterns) to verify you're using the same auth mechanism as existing endpoints. If existing endpoints use `requireAuth` and `requireRole('admin')`, use those exact functions — don't invent new ones.

**6.2 — Tenant isolation:**

For every database query this feature introduces:

```
Query: [describe what data is being fetched/modified]
  Tenant scoped: [yes — how? firmId from JWT? middleware injection? manual filter?]
  Verified by: [which layer enforces the scoping — middleware, service, repository?]
```

If the codebase has an existing tenant scoping pattern (e.g., a base repository that auto-applies firm ID), reference it and ensure this feature uses it. If the feature introduces a new query that doesn't go through the standard pattern, flag it as a risk.

**6.3 — Data exposure:**

For every API response this feature returns:

```
Endpoint: [METHOD /path]
  Fields returned: [list every field]
  Sensitive fields excluded: [confirm these are NOT in the response: internal IDs, other tenants' data, tokens, passwords, full user objects]
  DTO/serialiser: [which DTO or serialiser is used to control the response shape]
```

**6.4 — Input validation:**

For every user input this feature accepts:

```
Input: [field name]
  Source: [request body, query param, URL param, file upload]
  Validation: [type check, length limit, format regex, allowed values]
  Sanitisation: [HTML escaping, SQL parameterisation, etc.]
  Where validated: [which layer — route-level schema validation, service-level check, or both]
```

**6.5 — Data sovereignty (if applicable):**

If this feature introduces any new external API calls, third-party integrations, or data processing:

```
External service: [name]
  Data sent: [what data leaves the system]
  Data region: [where the external service processes data]
  Compliant: [yes/no — must be ap-southeast-2 or equivalent for AU law firm data]
```

### Section 7: File-by-File Plan

This is the core of the plan. List every file that needs to be created or modified, in implementation order (dependencies first).

For each file:

```
### [create/modify] `path/to/file.ts`

**What:** [1-2 sentences describing the change]
**Pattern:** Follow the pattern in `path/to/existing/similar/file.ts`
**Key details:**
- [Specific implementation detail that matters]
- [Specific implementation detail that matters]
**Depends on:** [other files from this plan that must be done first, or "none"]
```

**Implementation order rules:**

1. Database models/schemas first (everything else depends on them)
2. Service layer second (business logic)
3. API routes/controllers third (they call services)
4. Frontend components last (they call APIs)
5. Tests alongside each layer (not at the end)

**For every file**, verify:

- The file path follows the project's directory conventions
- The file name follows the project's naming conventions
- If creating a new file, reference an existing file of the same type as the pattern to follow
- If modifying an existing file, read it first to confirm the change makes sense in context

### Section 8: Test Plan

List every test that should be written, grouped by type. For each test, reference the existing test patterns in the codebase. Business logic tests should map directly to the input/output tables from Section 5 — every row in those tables becomes a test case.

**8.1 — Unit Tests:**

For every new or modified service function / utility function:

```
Test: [test name/description]
File: [path to test file, following existing naming convention]
What it validates: [one sentence — what business rule or logic path]
Pattern: Follow [path/to/existing/similar/test.ts]
```

Include at minimum: happy path, one error case, one edge case per function.

**8.2 — Business Logic Tests:**

For every function that implements a business rule, map tests directly from the Section 5 business rules tables. Every row in those tables becomes a test case:

```
Test: [test name/description]
File: [path to test file]
Source: [reference the specific row in the Section 5 table]
Scenarios:
  - Standard case: [input → expected output — from the table]
  - Boundary case: [input → expected output — from the table]
  - Invalid case: [input → expected error/behaviour — from the table]
```

If Section 5 defined state transitions, also test every valid and invalid transition:

```
Test: [transition name]
  - Valid: [current state] + [event] → [expected new state] + [expected side effects]
  - Invalid: [current state] + [wrong event] → [should remain in current state or error]
```

If Section 5 defined permission rules, test every cell in the permission matrix:

```
Test: [action] as [role]
  - Allowed: [role with ✓] can perform [action] → success
  - Denied: [role with ✗] cannot perform [action] → 403 or hidden
```

**8.3 — API Contract Tests:**

For every new or modified endpoint, test both the contract and the security design from Section 6:

```
Test: [test name/description]
File: [path to test file]
What it validates:
  - Response shape matches contract (all fields present, correct types)
  - Auth is enforced (401 without token — from Section 6.1)
  - Permission level enforced (403 for wrong role — from Section 6.1)
  - Tenant isolation holds (cannot access other tenant's data — from Section 6.2)
  - Sensitive fields excluded from response (from Section 6.3)
  - Input validation works (400 on invalid input — from Section 6.4)
  - Error responses match standard format
```

**8.4 — E2E Tests (if UI changes are involved):**

Map tests directly from the user flow defined in Section 4. Every step in the user flow becomes an assertion in the E2E test:

```
Test: [test name/description]
File: [path to Playwright test file]
Flow (from Section 4):
  1. [user action] → assert: [expected visible result from Section 4.1]
  2. [user action] → assert: [expected visible result]
UI state tests (from Section 4.2):
  - Loading: assert [loading indicator] is visible before data loads
  - Empty: assert [empty state message] when no data exists
  - Error: assert [error message] when API fails
Validation tests (from Section 4.3):
  - Submit with empty required field → assert [exact error message]
  - Submit with invalid input → assert [exact error message]
Permission tests (from Section 4.4):
  - User without permission → assert [feature is hidden/disabled]
```

### Section 9: Edge Cases and Risks

List everything that could go wrong at runtime:

- **Null/empty data:** What happens if [specific data] is missing or empty?
- **Permissions:** What happens if a user without the right role tries this?
- **Tenant isolation:** Could this accidentally expose data from another tenant?
- **Concurrent access:** Could two users doing this at the same time cause issues?
- **Partial failure:** If step 2 of 3 fails, what state is the data left in?
- **Scale:** What happens when there are 10,000 records? Is pagination needed?
- **Backward compatibility:** Does this change break any existing functionality?

For each risk, state how the plan handles it (or flag it as needing discussion).

### Section 10: Implementation Checkpoints

Break the implementation into natural checkpoints — moments where the developer should stop and verify before continuing. This prevents them from building an entire feature only to discover the foundation was wrong.

```
Checkpoint 1: Data model created and migration tested
  - Verify: schema matches plan, migration runs cleanly, existing data unaffected
  - Run: [test command for model tests]

Checkpoint 2: Service layer complete with unit tests
  - Verify: all service functions pass unit and business logic tests
  - Run: [test command for service tests]

Checkpoint 3: API endpoints complete with contract tests
  - Verify: all endpoints return correct shapes, auth works, tenant isolation holds
  - Run: [test command for API tests]

Checkpoint 4: Frontend complete with E2E tests (if applicable)
  - Verify: user flows work end-to-end in browser
  - Run: [Playwright command]
```

## Output Format

Structure the plan exactly as follows. The developer should be able to share this directly with a senior for review.

```
# Implementation Plan: [Feature Name]

## Approach
[3-5 sentences]

## Data Model Changes
[Schema definitions or "No database changes required"]

## API Contracts
[Endpoint definitions or "No API changes required"]

## User Experience
[User flow, UI states, validation/feedback, permissions in UI — or "No user-facing changes"]

## Business Rules
[Input → output tables, state transitions, permission matrices — or "No business logic"]

## Security Design
[Auth per endpoint, tenant isolation, data exposure, input validation]

## File-by-File Plan
[Ordered list of every file to create/modify]

## Test Plan
[Grouped test listings: unit, business logic, API contract, E2E]

## Edge Cases and Risks
[Risk list with mitigations]

## Implementation Checkpoints
[Checkpoint list with verification steps]

## Estimated Scope
Files to create: [N]
Files to modify: [N]

[List any ADRs created (as "proposed") or existing ADRs referenced during this plan.

```

## Important Behaviours

- **Reference real files, not abstract concepts.** Don't say "create a service." Say "create `src/services/templateService.ts` following the pattern in `src/services/documentService.ts`." Every file path must be verified against the actual codebase.
- **Follow existing patterns.** The plan should produce code that looks like it was written by the same team. If the codebase uses function-based services, don't propose class-based services. If routes are registered in a specific way, follow that way. Read the `docs/claude/` reference docs if they exist.
- **Respect existing ADRs.** If a past decision covers the area you're planning for, build on it — don't re-evaluate it. The only exception is when new constraints genuinely invalidate the previous decision, in which case flag it explicitly and get developer approval to supersede.
- **Only create ADRs for significant decisions.** Not every plan needs an ADR. The threshold is: 2+ genuinely different approaches were evaluated with real tradeoffs. If there was only one sensible approach, there's no decision to record.
- **ADRs are always created as "proposed."** The plan hasn't been reviewed yet — the developer and their reviewer decide whether the approach (and therefore the ADR) is accepted. Status updates to "accepted" when the PR merges. If the plan is rejected, delete the proposed ADR.
- **Be opinionated about the approach.** Don't present 5 options and ask the developer to choose. Recommend the best approach with reasoning. If there's a genuine tradeoff worth discussing, present 2 options max with clear tradeoffs and your recommendation.
- **Don't start implementing.** This skill produces a plan only. Do NOT create files, write code, or make changes. The only files this skill creates are ADRs in `docs/decisions/`.
- **Size the plan honestly.** If the plan reveals the ticket is much larger than expected (e.g., 20+ files, multiple migrations, 3 new API endpoints), say so explicitly. The developer needs to know before committing.
- **Make it reviewable in 5 minutes.** A senior should be able to scan the plan and spot issues quickly. The approach summary, data model, and API contracts are the most important sections for review. File-by-file details matter for the implementer, less for the reviewer.
